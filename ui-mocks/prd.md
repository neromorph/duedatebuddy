# PRD.md — Product Requirements Document
## Asset Tracker · Family Edition

**Version:** 1.0  
**Status:** MVP Definition  
**Stack:** Flutter 3.19+ · Supabase Cloud (ap-southeast-1) · FCM  
**Target users:** 2 (owner + spouse), expandable to small family  
**Timezone:** Asia/Jakarta (WIB, UTC+7)

---

## 1. Product Overview

Asset Tracker is a private, offline-first mobile application for iOS and Android that helps a family track their physical and financial assets — including billing due dates, document expiry (STNK, BPKB, insurance, certificates), and renewal reminders. Data is synced in real-time between family members via Supabase Cloud.

### 1.1 Core Value Proposition

- Never miss a renewal, payment, or document expiry again.
- All asset information, documents, and deadlines in one place.
- Works on both family members' phones, always in sync.
- Works offline; syncs when internet is available.

### 1.2 Non-Goals (v1.0)

- Not a financial accounting or bookkeeping app.
- No public sharing, social features, or multi-tenancy.
- No web app — mobile only.
- No payment processing or integration with banks.
- No AI features in v1.0.

## 2. User Roles

| Role | Description |
|---|---|
| **Owner** | Creates the family group, manages members, has full CRUD on all assets |
| **Member** | Invited by Owner, has full CRUD on assets within the family group |

Both roles share identical UI and capabilities in v1.0. Role distinction exists in the data model for future expansion.

## 3. Feature Specification

### 3.1 Authentication

**Requirements:**
- Email + password registration and login.
- Google OAuth login (one-tap on Android, standard on iOS).
- PKCE auth flow for mobile security.
- Auto-login on app reopen if session is valid.
- Biometric lock (fingerprint/Face ID) as a secondary lock after login.
- App locks after 5 minutes of backgrounding; biometric required to re-enter.
- Logout clears local session and navigates to login screen.

**Supabase:** `auth.users` table, managed by Supabase Auth. `users` table in public schema mirrors auth user with additional profile fields.

---

### 3.2 Family Group

**Requirements:**
- On first login, user either creates a new family group or joins via an invite code.
- Invite code is a 6-character alphanumeric code generated by Supabase Edge Function.
- Invite code expires after 24 hours.
- Max 6 members per family group (v1.0).
- Owner can remove a member from the group.
- Leaving a group does not delete shared assets — they remain in the family group.

**Schema:**
```sql
create table family_groups (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  created_by uuid references auth.users(id),
  created_at timestamptz default now()
);

create table family_memberships (
  id uuid primary key default gen_random_uuid(),
  family_group_id uuid references family_groups(id) on delete cascade,
  user_id uuid references auth.users(id) on delete cascade,
  role text not null default 'member' check (role in ('owner', 'member')),
  joined_at timestamptz default now(),
  unique(family_group_id, user_id)
);

create table invite_codes (
  id uuid primary key default gen_random_uuid(),
  family_group_id uuid references family_groups(id) on delete cascade,
  code text not null unique,
  created_by uuid references auth.users(id),
  expires_at timestamptz not null,
  used_at timestamptz,
  used_by uuid references auth.users(id)
);
```

### 3.4 Assets

An asset is a family-owned record created from a template or from scratch.

**Requirements:**
- Create asset from a template (fields pre-populated) or from scratch (custom).
- Edit any field on an existing asset.
- Archive an asset (soft delete — hidden from main list, accessible in archive).
- Hard delete requires explicit confirmation dialog.
- Assets belong to the family group — both members can view and edit all assets.
- Each asset can have multiple extension records and multiple file attachments.

**Schema:**
```sql
create table assets (
  id uuid primary key default gen_random_uuid(),
  family_group_id uuid references family_groups(id) on delete cascade,
  template_id uuid references asset_templates(id),  -- nullable for custom assets
  created_by uuid references auth.users(id),
  name text not null,
  category text not null,
  icon_name text not null,
  notes text,
  custom_fields jsonb default '{}',
  is_archived boolean default false,
  sync_status text default 'synced' check (sync_status in ('synced','pending_create','pending_update','pending_delete')),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

### 3.5 Extension Records (Billing & Document Expiry)

Each asset can have multiple tracked items — a vehicle might have an STNK expiry, a KIR expiry, and a monthly insurance billing all tracked separately.

**Extension types:**
- `document` — a document with a physical expiry date (STNK, passport, IMB, certificate).
- `billing` — a recurring payment (monthly, quarterly, annual).
- `reminder` — a one-off reminder (service schedule, inspection date).

**Requirements:**
- Each extension record has an expiry or due date.
- User sets a reminder window: 7, 14, 30, 60, or 90 days before expiry, or custom days.
- On billing type: set cycle (monthly, quarterly, semi-annual, annual), amount, and currency (IDR default).
- After expiry date passes: status changes to `expired`. System does not auto-renew.
- User manually marks as renewed, entering the new expiry date.
- Renewing archives the old record and creates a new one (preserving history).
- Status values: `active`, `expiring_soon` (within reminder window), `expired`, `renewed`.

**Schema:**
```sql
create table extension_records (
  id uuid primary key default gen_random_uuid(),
  asset_id uuid references assets(id) on delete cascade,
  family_group_id uuid references family_groups(id),
  type text not null check (type in ('document','billing','reminder')),
  label text not null,
  expiry_date date not null,
  remind_days_before int not null default 30,
  billing_cycle text check (billing_cycle in ('monthly','quarterly','semi_annual','annual','one_time')),
  amount numeric(15,2),
  currency text default 'IDR',
  status text not null default 'active' check (status in ('active','expiring_soon','expired','renewed')),
  renewed_from uuid references extension_records(id),  -- links renewal chain
  sync_status text default 'synced' check (sync_status in ('synced','pending_create','pending_update','pending_delete')),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

### 3.6 File Attachments

Users can attach photos or PDFs to any extension record (e.g., scan of STNK, insurance policy PDF).

**Requirements:**
- Accepted types: `jpg`, `jpeg`, `png`, `heic`, `pdf`.
- Max file size: 10 MB per file.
- Max files per extension record: 5.
- Files stored in Supabase Storage, private bucket named `asset-documents`.
- File access via signed URL with 1-hour expiry.
- Thumbnail generated client-side for image previews.
- Files stored locally (cached) for offline viewing.
- Delete file: removes from Supabase Storage and local cache.

**Schema:**
```sql
create table attachments (
  id uuid primary key default gen_random_uuid(),
  extension_record_id uuid references extension_records(id) on delete cascade,
  family_group_id uuid references family_groups(id),
  uploaded_by uuid references auth.users(id),
  file_name text not null,
  file_type text not null,
  file_size_bytes int not null,
  storage_path text not null,  -- path inside Supabase Storage bucket
  local_cache_path text,       -- device-local cache path
  sync_status text default 'synced',
  uploaded_at timestamptz default now()
);
```

### 3.7 Notifications

**Local notifications (on-device, no server needed):**
- Scheduled at the reminder window before expiry (e.g., 30 days before STNK expires).
- Re-scheduled whenever the extension record is created, edited, or renewed.
- Cancelled when the extension record is deleted or marked renewed.
- Each notification deep-links into the specific extension record.
- All notifications use `Asia/Jakarta` timezone for scheduling.

**Push notifications (via FCM — cross-device):**
- Sent by a Supabase Edge Function cron job running daily at 08:00 WIB.
- Targets all family members of the group for assets entering the expiry window.
- Push payload includes: asset name, extension label, days remaining, deep-link URL.
- FCM tokens stored per user device in the `user_devices` table.
- Handles token refresh automatically via `firebase_messaging` callbacks.

**Schema:**
```sql
create table user_devices (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users(id) on delete cascade,
  fcm_token text not null,
  platform text not null check (platform in ('ios','android')),
  updated_at timestamptz default now(),
  unique(user_id, fcm_token)
);
```

**Notification channels (Android):**
- `expiry_critical` — expires in ≤7 days (high priority, sound on)
- `expiry_warning` — expires in 8–30 days (default priority)
- `expiry_info` — expires in 31–90 days (low priority, silent)

### 3.8 Dashboard

**Requirements:**
- Default landing screen after login.
- Summary metrics at top: total assets, expiring this month, expired (overdue).
- "Expiring soon" section: extension records within 30 days, sorted by soonest first.
- "All assets" section: grouped by category, each showing asset name + next expiry badge.
- Color-coded expiry badge: red (≤7 days), amber (8–30 days), green (>30 days), gray (no expiry tracked).
- Search bar: searches across asset names, notes, and custom fields.
- Filter: by category, by status (active, expiring soon, expired).
- Pull-to-refresh triggers sync.

### 3.10 Expiry Engine Logic

The expiry engine runs locally on-device to compute statuses and schedule notifications.

```
expiry_engine rules (all times in Asia/Jakarta):

1. Get today's date in Asia/Jakarta timezone.
2. For each active extension_record:
   a. days_remaining = expiry_date - today (in full days)
   b. if days_remaining < 0 → status = 'expired'
   c. if days_remaining <= remind_days_before → status = 'expiring_soon'
   d. else → status = 'active'
3. Schedule/reschedule local notification:
   a. trigger_date = expiry_date - remind_days_before (at 08:00 WIB)
   b. if trigger_date is in the past and status is expiring_soon → notify immediately
   c. else → schedule for trigger_date
4. Run on: app launch, app foreground, after any extension_record save.
```

## 4. Data Sync Architecture

```
Device (Drift local DB)
    │
    ├─ READ: always from local DB (instant, offline-safe)
    │
    └─ WRITE:
        1. Write to local DB with sync_status = 'pending_create/update/delete'
        2. SyncService detects pending records
        3. On network available → push to Supabase
        4. On success → update sync_status = 'synced'
        5. On failure → retry with exponential backoff (max 5 retries)

Supabase Realtime
    │
    └─ SUBSCRIBE: to changes in assets, extension_records, attachments
                  filtered by family_group_id
        → On change received → upsert into local Drift DB
        → UI reactively updates via Riverpod providers
```

**Conflict resolution:** Last-write-wins based on `updated_at` timestamp. This is acceptable for a 2-person family app with low concurrent edit probability.

├── Auth flow (GoRouter guard)

### 5.2 Theme

- Material 3 with custom color scheme.
- Primary color: deep teal (`#006874`) — conveys trust and clarity.
- Support light and dark mode from day one.
- Font: `Plus Jakarta Sans` (Google Fonts) — modern, readable, works well with Indonesian text.
- All spacing on an 8dp grid.

### 5.3 Empty States

Every list screen must have a meaningful empty state with an illustration, a title, and a CTA button. Never show a blank screen.

### 5.4 Offline Indicator

A persistent subtle banner at the top of the screen when the device is offline. It must not block content — overlay a thin colored strip with text "Mode offline — perubahan akan disinkronkan saat online."

## 7. MVP Scope (Phase 1)

The following must be complete and working before the app is considered MVP:

- [ ] Auth: email/password login, register, logout, biometric lock
- [ ] Family group: create, join via invite code
- [ ] Asset templates: all 10 pre-built templates seeded
- [ ] Assets: create (from template + custom), view list, view detail, edit, archive
- [ ] Extension records: create, view, edit, mark-as-renewed, delete
- [ ] Attachments: upload photo/PDF, view, delete
- [ ] Dashboard: summary metrics, expiring soon list, all assets grouped
- [ ] Local notifications: scheduled reminders per extension record
- [ ] Offline-first: full CRUD works without internet, syncs on reconnect
- [ ] Realtime sync: changes on one device appear on the other within 5 seconds
- [ ] Dark mode: all screens support light and dark

**Not in MVP (Phase 2+):**
- FCM push notifications (use local notifications only for MVP)
- Calendar view tab
- Statistics/charts screen
- Export to PDF
- WhatsApp reminder integration
- Widget (home screen widget showing next expiry)
